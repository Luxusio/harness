## Problem Statement

Developers using Claude Code for serious multi-agent work face three compounding failures:
1. **Manual agent document management** — project constraints must be copy-pasted to every agent invocation
2. **Manual verification** — no automatic QA of agent outputs against project rules
3. **Document integrity violations** — agents freely ignore CLAUDE.md and other project constraints

Without structured enforcement, multi-agent AI development collapses into copy-paste chaos with unpredictable outputs.

- Copy-paste workaround exists, confirming the problem is real and people tolerate pain to work around it
- Subagent noted: Pain #3 (constraint violation) is a *correctness failure*, not friction — different category requiring an enforcement runtime, not a reminder system

## Status Quo

Current workaround: copy-paste all agent instructions per invocation.

Cost breakdown:
- Time: repeated manual context injection per agent
- Inconsistency: different agents receive different instructions → divergent outputs
- No rollback: violations discovered after the fact, not blocked at commit

**Narrowest wedge:** The critic-as-gatekeeper. One critic agent that blocks commits when outputs violate CLAUDE.md rubric. Everything else (task lifecycle, DOC_SYNC, team coordination) can come after.

## Approaches Considered

### Approach A: CLI + Git Hook Minimal
Single bash pre-commit hook. CLAUDE.md as rubric. LLM call on staged diff. Blocks git commit on FAIL.
- Effort: S (~1 day human / ~1 hour CC+gstack) | Risk: Low | Completeness: 4/10
- Skip: task lifecycle, multi-agent coordination, MCP
- Good for: rapid demand validation before investing in B

### Approach B: Claude Code MCP Plugin (SELECTED)
Full Claude Code MCP integration. Components: task lifecycle state machine, critic gates (plan/runtime/document), DOC_SYNC writer, team agent coordinator, TASK_STATE.yaml persistence.
- Effort: XL (~3 months for a 2-person team / ongoing with CC+gstack) | Risk: High | Completeness: 9/10
- Moat: the only tool that enforces full plan→implement→verify→close lifecycle inside Claude Code
- "Deepest integration" defined as: only tool with critic gates at all three workflow phases (plan, runtime, doc) AND team agent orchestration AND task state persistence
- Risk mitigation: MCP SDK abstracted behind a thin adapter layer; if SDK breaks, adapter swapped without rewriting critics or task logic

### Approach C: Enforcement SDK
harness-plugin as npm/pip package. Other projects import critic+enforcement logic as a library.
- Effort: M (~3 weeks human / ~3 hours CC+gstack) | Risk: Med | Completeness: 7/10
- npm downloads as demand signal
- Requires headless refactor (decouple from CLAUDE.md file convention)

## Technical Architecture (Approach B)

Components (current v3 state + planned):
```
Claude Code (user session)
  └── harness MCP server (this repo)
       ├── task_start / task_context / task_close  ← task lifecycle
       ├── critic-plan (PLAN.md evaluation)
       ├── critic-runtime (evidence bundle QA)
       ├── critic-document (DOC_SYNC verification)
       ├── team_bootstrap / team_dispatch          ← multi-agent coordination
       └── TASK_STATE.yaml                         ← persistent state
```

MCP SDK adapter layer (planned — not yet built): will isolate Anthropic API surface. Critics are stateless functions: input=(artifact path, rubric), output=(PASS|FAIL, evidence). Task state machine in YAML — no database dependency.

## Security & Trust Model

harness-plugin sits at a trust boundary: it wraps agent output and can block commits.

- Critics are read-only LLM calls — no write access to project files during evaluation
- TASK_STATE.yaml is git-tracked — tamper is visible in diff
- If critic is wrong (false FAIL): user can `--force` bypass with explicit acknowledgment logged
- If harness crashes mid-task: TASK_STATE.yaml preserves last known state; task can be resumed
- Threat: malicious CLAUDE.md rubric could cause always-PASS. Mitigation: rubric changes require user review (not agent-writable by default)
