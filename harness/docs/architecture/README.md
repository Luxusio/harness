## System Boundaries

- `plugin/` — all plugin runtime code (skills, agents, hooks, scripts) — confirmed
- `harness/` — repo-local operating state (generated per-project by setup) — confirmed
- `.claude-plugin/` — marketplace metadata — confirmed

## Key Patterns

- Hidden workflow SKILL.md files are internal procedure documents: SKILL.md contains step-by-step instructions read by the orchestrator, not executable code and not user-invocable commands — confirmed
- User-invocable command surface is limited to two public commands: `/harness:setup` (bootstrap) and `/harness:validate` (diagnostic). All other workflows are hidden internal procedures — confirmed
- Agent delegation: orchestrator reads SKILL.md procedure doc then delegates to specialist agents via Agent tool — confirmed
- Template-based setup: `skills/setup/templates/` contains all scaffolding templates with `{{PLACEHOLDER}}` syntax — confirmed
- Approval gate source of truth: `harness/policies/approvals.yaml` is the enforcement gate the orchestrator checks before touching sensitive areas. `harness/manifest.yaml` `risk_zones` is descriptive context only — confirmed

## Shared Compiled Memory

The repository uses a two-tier memory system — confirmed:

### Source of truth (durable, human-editable)
- `harness/docs/*` — constraints, decisions, requirements, architecture, runbooks
- `harness/state/*` — recent decisions, unknowns
- `harness/policies/*` — approvals, memory policy

### Compiled index (generated, machine-optimized)
- `harness/memory-index/` — deterministic JSON compilation of all durable sources
- Generated by `harness/scripts/build-memory-index.py` (no LLM involvement)
- Committed to git — all team members share the same retrieval advantage
- **Never manually edited** — always regenerated from source
- Scope indexing is active: `by-domain/` and `by-path/` are populated on each rebuild
- Temporal relations are populated from ADR supersession chains in `relations` fields

### Local overlay (per-session, gitignored)
- `.harness-cache/memory-overlay/` — uncommitted working memory
- Built by `harness/scripts/build-memory-overlay.sh` from current session state
- Merged at query time via `--include-overlay` flag, overrides compiled index for same subject
- Cleaned up when changes are committed; gitignored — per-session only

### Retrieval flow
```
query → compiled index (fast prefilter) → top candidates
     → optional: local overlay merge (session-specific overrides)
     → raw source verification (top 4 files)
     → optional: agentic fan-out (temporal/conflict queries only)
     → single authoritative memory pack → orchestrator
```

### Temporal relations — confirmed
Records track supersession chains through `relations`:
- `supersedes`: this record replaces an older one
- `extends`: this record builds on another
- `resolves`: this record answers an open question
- `conflicts_with`: this record contradicts another (needs resolution)

Superseded records are not deleted — they remain in `timeline/` for historical queries.

### Regression tests
```bash
bash harness/scripts/test-memory-index.sh
```
Runs 5+ automated checks: schema validity, determinism, scope coverage, temporal relations, query output format.
